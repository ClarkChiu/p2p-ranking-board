## Why

我想被動追蹤海盜灣(The Pirate Bay)榜單,有新片/熱門更新時收到通知,再自行決定要不要下載 —— 而不是每天手動去翻網站。現有的 p2pscout(專案 A)只能「我主動下關鍵字才搜尋」,缺少「榜單有變動就推播給我」這條被動觸發鏈。本專案補上這段,並在我核准後輸出選定項目的磁力連結,讓我以管道交給 p2pscout 做健康度驗證與下載。

## What Changes

- 新增定期抓取海盜灣榜單的能力(第一版資料源採用 apibay 這個官方 JSON 後端,而非爬 HTML)。
- 新增榜單快照持久化與差異比對:辨識「新進榜」「排名變動」「種子數明顯變動」的項目。
- 新增變動通知:把差異結果推播給我;通知管道設計成可插拔(第一版預設一種管道即可)。
- 新增磁力連結輸出:我核准某項目後(用一個 CLI 子命令給識別碼),把對應磁力連結印到標準輸出,僅此而已。驗證與下載交給別的程式以管道串接,本工具不執行也不相依。
- 排程不內建:第一版以「一次性 CLI 執行 + 外部排程器(cron 等)」達成定期輪詢,狀態存在本機快照檔。

## Capabilities

### New Capabilities

- `ranking-source`: 從海盜灣取得指定榜單(類別 / 範圍),把回應正規化成統一的榜單條目(名稱、資訊雜湊、磁力連結、類別、種子節點數、排名、抓取時間)。封裝資料源細節,日後可換不同來源或鏡像。
- `change-detection`: 載入上一次快照、與本次榜單比對、產生差異(新進、離榜、排名升降、種子數變動),並原子性寫回新快照。第一版以本機檔案儲存快照。
- `update-notification`: 把差異結果格式化並透過可插拔的通知管道送出;支援設定門檻(例如只通知新進榜或排名躍升 N 名以上),避免雜訊。
- `magnet-resolve`: 把我核准的條目識別碼還原成磁力連結,輸出到標準輸出,僅此而已。健康度驗證與下載由另一支程式(p2pscout)負責,由呼叫者以 shell 管道串接(`get <id> | p2pscout get --magnet -`);ranking-board 不相依、也不認識該程式。
- `ranking-list`: 不經比對,直接列出當前各追蹤類別的前 N 名榜單條目,供使用者隨時查看「現在熱門有什麼」(與比對通知鏈互補)。

## Impact

- 新專案,獨立於 p2pscout(專案 A)。兩者唯一介面是「磁力連結字串」,由呼叫者以 shell 管道串接;雙方互不相依、互不認識。
- 外部相依:海盜灣 apibay 端點(需於 design 階段確認確切 URL 與回應結構);外部代理 / 排程器。**不**相依任何下載程式。

### 已拍板決策(第一版)

- **實作語言**:Go,與 p2pscout 一致。
- **通知管道**:第一版走標準輸出 / 本機 log;通知管道介面仍保留,日後可加 Telegram 等。
- **排程觸發**:程式本身維持「無狀態一次性執行」(抓取 → 比對 → 寫回快照 → 把差異以結構化格式印出),不內建常駐排程;由外部代理(使用者採用 Hermes Agent)定時呼叫。判斷「哪些變動值得提醒」這類需要語意判斷的工作,留在外部代理的大型語言模型層,**不**寫進 Go 程式(維持 Go 端純為確定性轉換)。

### 已選定類別(第一版)

追蹤這些 apibay 類別代碼:`207`(HD 電影)、`208`(HD 影集)、`301`(Windows)、`303`(Linux/UNIX)、`401`(PC 遊戲)、`601`(電子書)。每個類別各有一張 top100 榜單,合併後一起比對。(原本含 `300`「應用程式」,但實測它回傳與 `301` 相同的 Windows 條目,屬冗餘,已移除。)

### 待 design 確認

- apibay 確切端點與回應欄位(`q.php` 與 `precompiled/data_top100_<cat>.json` 已實測可用)、快照檔格式與位置、通知門檻策略。
